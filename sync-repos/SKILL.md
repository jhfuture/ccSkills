# Git 仓库批量同步 Skill

这个 skill 帮助用户批量更新当前目录及一级子目录下的所有 git 仓库。

## 工作流程

### 第一步：扫描并识别仓库

运行 `sync_repos.py` 脚本的扫描模式:

```bash
python scripts/sync_repos.py --scan-only
```

脚本会输出找到的所有 git 仓库列表,格式如下:

```
发现 3 个 Git 仓库:
1. ./repo1 (分支: main)
2. ./subdir/repo2 (分支: develop)  
3. ./repo3 (分支: feature/new)
```

### 第二步：向用户确认

将扫描结果展示给用户,并询问:

"发现了以上 X 个 Git 仓库,是否继续同步更新?"

等待用户明确确认(如"是"、"继续"、"确认"等)后再进行下一步。

### 第三步：执行同步

用户确认后,运行完整的同步命令:

```bash
python scripts/sync_repos.py
```

### 第四步：生成报告

脚本会输出 JSON 格式的结果。根据 JSON 结果生成用户友好的报告。

## 报告格式

报告应该清晰地分为三个部分:

### 成功更新的仓库
对于每个成功更新的仓库,显示:
- 仓库路径
- 拉取的新提交数量
- 变更的文件列表(新增/修改/删除)

示例:
```
✓ ./repo1 (main)
  拉取了 3 个新提交
  变更文件:
  - src/main.js (修改)
  - README.md (修改)
  - tests/new.test.js (新增)
```

### 跳过的仓库
对于有问题无法更新的仓库,显示:
- 仓库路径
- 跳过原因(未提交修改/合并冲突/其他错误)
- 建议用户手动处理

示例:
```
⚠️ ./repo2 (develop)
  跳过原因: 有未提交的修改
  请手动处理此仓库
```

### 已是最新的仓库
对于已经是最新状态的仓库:
```
→ ./repo3 (main)
  已是最新,无需更新
```

## 统计摘要

在报告末尾添加统计信息:
```
总计: 5 个仓库
- 成功更新: 2 个
- 已是最新: 2 个  
- 跳过: 1 个
```

## 错误处理原则

- 如果仓库有未提交的修改,跳过并报告
- 如果 git pull 失败(如合并冲突),跳过并报告错误信息
- 如果仓库没有远程分支,跳过并说明原因
- 继续处理其他仓库,不要因为一个仓库失败就停止

## 注意事项

- 只扫描当前目录和一级子目录,不递归深层目录
- 每个仓库拉取其当前所在分支的最新代码
- 不自动切换分支
- 不自动处理冲突或 stash 修改
